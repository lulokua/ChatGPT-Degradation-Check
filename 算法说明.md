# ChatGPT 降智检测 Pro 算法说明

## 文档范围

- 对应脚本版本：`3.4.0`
- 对应算法版本：`v5`
- 核心实现文件：[ChatGPT降智检测.user.js](./ChatGPT降智检测.user.js)

这份文档面向两类读者：

- 普通用户：理解脚本为什么这样判断
- 进阶用户：理解评分由哪些信号组成、各信号如何融合

---

## 1. 算法目标

这套算法要解决的不是“读取一次 IP 分数”，而是：

> 结合当前 IP 质量、最近一段时间的历史行为，以及 ChatGPT 返回的 PoW 难度信号，判断当前会话是否存在降智风险。

它的输出不是单一结论，而是一组可解释指标：

- `compositeRisk`：综合风险评分，范围 `0-100`
- `stability`：稳定性评分，越高越稳定
- `confidence`：置信度，表示当前判断有多可信
- `volatility`：波动指数，越高代表越不稳定
- `trend`：趋势方向
- `verdict`：最终文字结论

---

## 2. 设计原则

这套算法基于三个原则：

### 2.1 不迷信单次分数

一次 `IPPure` 风控分只能说明“这一刻”的情况，不能说明这个节点最近是否持续恶化，也不能说明它是不是一个不稳定的共享节点。

### 2.2 不迷信单次 PoW

`PoW difficulty` 很有价值，但它本身也会受问题复杂度、页面状态、请求路径等因素影响。  
因此，PoW 更适合作为“辅助证据链”，不适合单独判死刑。

### 2.3 更看重趋势和一致性

真正有价值的判断，不是“这次高不高”，而是：

- 最近是不是在恶化
- 历史是不是稳定
- 当前 IP 自身是不是长期偏差
- PoW 信号和 IP 风险信号是否一致

---

## 3. 输入数据来源

算法同时使用三类输入。

### 3.1 当前实时 IP 信号

来源：

- `https://chatgpt.com/cdn-cgi/trace`
- `https://my.ippure.com/v1/info`

用途：

- 获取浏览器访问 ChatGPT 时的真实出口 IP
- 查询当前 IP 的 `IPPure` 风控分
- 获取 ASN、地区、住宅/机房等基础网络属性

核心字段：

- `currentScore`
- `currentIP`

### 3.2 本地历史 IP 风险记录

脚本会把每次检测结果保存在浏览器本地 `localStorage` 中。

历史记录字段：

- `score`：本次 IP 风控分
- `ts`：时间戳
- `ip`：检测时的出口 IP

用途：

- 观察一段时间内的风险趋势
- 识别是否频繁切换 IP
- 判断是否存在锯齿波动、连续恶化等模式

### 3.3 ChatGPT PoW 难度记录

来源：

- `chat-requirements`
- `sentinel`
- `chat-requirements/prepare`

脚本会监听响应中的 `proofofwork.difficulty`，并记录到本地。

PoW 历史字段：

- `difficulty`
- `digits`
- `value`
- `ts`
- `required`
- `source`

用途：

- 反映当前对话入口质量
- 作为 IP 风险结论的辅助验证
- 在信号一致时提升置信度，在信号冲突时降低置信度

---

## 4. 数据采集流程

完整采集流程如下：

1. 通过 `cdn-cgi/trace` 获取 ChatGPT 实际看到的出口 IP
2. 使用该 IP 调用 `IPPure` 获取当前基础风险分
3. 将每次 IP 风险检测结果写入本地历史
4. 监听页面内 `chat-requirements` / `sentinel` 响应
5. 抽取 `PoW difficulty` 并写入本地 PoW 历史
6. 每次展示面板时，重新用“当前样本 + 本地历史”计算综合风险

这意味着：

- 脚本是“持续积累型”工具
- 样本越多，判断越稳定
- 不是只给你一个静态分数，而是持续修正结论

## 5. IP 风险历史维度

这是算法的主干部分。

### 5.1 历史清洗

历史数据会先做标准化：

- 强制把 `score` 转为数值
- 把分数裁剪到 `0-100`
- 丢弃非法时间戳
- 按时间升序排序

目的：

- 避免脏数据污染趋势判断
- 保证后续时序计算可靠

### 5.2 会话聚合

短时间内连续刷新不应该被当作多个独立样本。  
因此算法会按“会话”聚合历史记录。

合并条件：

- 同一个 IP
- 两次记录间隔小于 `30 分钟`

每个会话的得分不是简单取最后一次，而是：

```text
sessionScore =
  55% * lastScore
  + 25% * sessionAvg
  + 20% * sessionUpperQuartile
```

这么做的原因：

- 保留最新状态
- 不忽略会话中间出现的高风险尖峰
- 降低频繁刷新带来的噪声

### 5.3 基础统计特征

会话化之后，算法提取以下统计量：

- `avgScore`：历史平均分
- `stdDev`：标准差
- `robustStd`：基于 `MAD` 的稳健波动估计
- `q75`：上四分位数，反映高风险尾部

这些特征用于回答：

- 这个节点长期平均风险高不高
- 历史是稳定分布还是离散严重
- 有没有高风险尾部拖累

### 5.4 时间衰减特征

算法不会把一周前的数据和一分钟前的数据看得一样重。  
因此使用指数衰减加权均值：

```text
weightedAvg = Σ(score_i * weight_i) / Σ(weight_i)
weight_i = exp(-lambda * age_i)
half-life = 24h
```

也就是说：

- 越新的记录，权重越高
- 越旧的记录，影响越弱

这样可以让最近状态更快地反映到综合结果里。

### 5.5 趋势特征

算法同时看三类趋势信号：

- `EMA(5)` 与 `EMA(15)`
- 最近窗口与更早窗口均值差 `trendDelta`
- 最近最多 `6` 个会话的线性斜率 `slopeRecent`

趋势总信号：

```text
trendSignal =
  trendDelta * 0.55
  + emaCross * 0.9
  + slopeRecent * 3.5
```

其中：

- `emaCross = emaShort - emaLong`
- `trendSignal > 0`：偏恶化
- `trendSignal < 0`：偏改善

最终会映射为：

- 双线确认恶化
- 趋势恶化
- 双线确认改善
- 趋势改善
- 短期恶化信号
- 短期改善信号
- 基本持平

### 5.6 波动与异常模式

算法会检测多种“不稳定节点特征”。

#### 1. 区间跳跃

- `zoneJumps`：风险区间切换次数
- `bigJumps`：相邻会话分差大于 `15` 的次数

这类指标用来识别：

- 节点是否经常在不同风险段来回跳
- 是否存在突然失稳或突然恢复

#### 2. 锯齿波动

如果最近几次变化频繁出现“高低反转”，且波动幅度明显，算法会识别为锯齿波动。

这类模式通常意味着：

- 共享节点
- 轮转节点
- 不稳定出口
- 网络链路状态反复变化

#### 3. 阶梯恶化

如果多个会话持续递增，说明风险正在一层一层抬高。

这比“一次性高分”更危险，因为它代表趋势恶化。

### 5.7 持续性风险特征

算法会额外看“持续高风险”：

- `recentHighRiskRatio`：最近窗口高风险占比
- `currentHighRiskStreak`：当前连续高风险会话数
- `maxHighRiskStreak`：历史最长高风险连续段

意义：

- 防止“刚刚短暂回落”就被误判为安全
- 对持续恶化更敏感

### 5.8 当前 IP 独立历史

算法不仅看“全部历史”，还单独看“当前这个 IP 自己的历史”：

- `cipAvg`：当前 IP 历史均值
- `cipRecentAvg`：当前 IP 近期均值

这能把下面两种情况区分开：

- 整体样本混杂，因为你最近频繁换节点
- 当前这个 IP 本身长期偏差

## 6. PoW 维度

PoW 是这版算法的重要增强。

### 6.1 为什么引入 PoW

单纯看 IP 风控分，有时会遇到：

- 当前分数还行，但体验已经开始异常
- 历史样本不够多，难以下结论
- 节点刚切换，IP 数据尚未积累充分

这时候 `PoW difficulty` 可以提供一个非常有用的侧证：

- 难度长期偏低，往往意味着入口质量较弱
- 难度长期偏高，通常说明入口质量较稳

但需要强调：

> PoW 只是辅助证据，不是单独判定依据。

### 6.2 PoW 标准化

原始 `difficulty` 是十六进制字符串。  
算法会先做标准化：

- 去空格
- 去 `0x`
- 只保留十六进制字符
- 去除前导零

随后提取两个值：

- `digits`：difficulty 的十六进制位数
- `value`：difficulty 转为十进制后的数值

### 6.3 PoW 强度映射

为了让 PoW 可以和其他指标融合，算法先把 difficulty 映射到一个 `0-100` 的强度空间：

```text
powStrength = clamp(((log16(value + 1) - 2) / 3) * 100, 0, 100)
```

直观理解：

- difficulty 越大，`powStrength` 越高
- difficulty 越小，`powStrength` 越低

### 6.4 采样窗口

PoW 历史默认使用：

- 最近 `12 小时` 的样本
- 如果 12 小时内没有样本，回退到最近 `8` 个样本

同时定义：

- `powFresh`：最近一次 PoW 是否在 `2 小时` 内

这样做是为了优先相信“最近的入口状态”。

### 6.5 PoW 风险提取

算法会从最近窗口提取：

- `powCurrentDigits`
- `powCurrentStrength`
- `powLowRatio`：低难度占比，`digits <= 3`
- `powHighRatio`：高难度占比，`digits >= 5`
- `powTrend`：最近 PoW 强度趋势
- `powSampleCount`

### 6.6 PoW 风险公式

PoW 风险 `powRisk` 的核心计算如下：

```text
powRisk =
  100 - (currentStrength * 0.58 + medianStrength * 0.42)
  + clamp(lowRatio * 20 - highRatio * 8, -8, 12)
  + clamp(digitStd * 6, 0, 10)
  - clamp(strengthTrend * 0.18, -8, 8)

if currentDigits <= 3: powRisk += 6
if currentDigits >= 5: powRisk -= 4
```

直观解释：

- 当前难度越低，风险越高
- 最近窗口低难度占比越高，风险越高
- 最近窗口高难度占比越高，风险越低
- 如果 PoW 强度在改善，风险会被适当拉低
- 如果 PoW 位数波动太大，也会增加不稳定惩罚

### 6.7 PoW 关键阈值

当前版本采用下面的经验阈值：

- `digits <= 3`：明显偏弱
- `digits >= 5`：明显偏强

辅助状态：

- `powSevere`：
  - `powRisk >= 68`
  - 或当前 `digits <= 3` 且 `powLowRatio >= 50%`

- `powPositive`：
  - `powRisk <= 30`
  - 且当前 `digits >= 5`
  - 且 `powHighRatio >= 30%`

## 7. 综合风险分合成

这是最终最核心的一步。

### 7.1 基础合成项

新版综合风险分以以下部分为基础：

```text
baseCompositeRisk =
  weightedAvg * 0.24
  + currentScore * 0.24
  + cipAvg * 0.15
  + cipRecentAvg * 0.12
  + recentAvg * 0.14
  + q75 * 0.11
```

这代表：

- 当前实时风险
- 最近一段时间趋势
- 当前 IP 独立历史
- 高风险尾部

共同决定初始风险底盘。

### 7.2 IP 历史维度修正项

在基础分上，算法还会叠加：

```text
trendAdjustment = clamp(trendSignal * 1.25, -8, 12)
persistencePenalty = recentHighRiskRatio * 10 + min(currentHighRiskStreak * 2.5, 10)
patternPenalty = sawtooth ? 6 : 0 + stair ? 8 : 0 + bigJumpRate > 0.3 ? 4 : 0
peakPenalty = clamp((recentPeakScore - recentAvg) * 0.18, 0, 4)
recoveryBonus = 4   // 仅在明显恢复时触发
```

说明：

- 趋势恶化会提高风险
- 连续高风险会提高风险
- 锯齿/阶梯/大跳变会提高风险
- 近期尖峰会提高风险
- 如果当前状态明显恢复，会给一点减分奖励

### 7.3 PoW 融合项

PoW 不直接替代基础分，而是作为辅助修正项：

```text
powImpactWeight =
  0                    // 没有 PoW 样本
  0.22 / 0.12 / 0.08   // 根据样本数量与新鲜度选择

powAdjustment = clamp((powRisk - 45) * powImpactWeight, -8, 12)
powPatternPenalty = 6  // 当前位数 <= 3 且低难度占比 >= 50%
powRecoveryBonus = 4   // 当前位数 >= 5 且高难度占比 >= 35%
```

最终：

```text
compositeRisk =
  baseCompositeRisk
  + trendAdjustment
  + persistencePenalty
  + patternPenalty
  + peakPenalty
  + powAdjustment
  + powPatternPenalty
  - recoveryBonus
  - powRecoveryBonus
```

最后再裁剪到 `0-100`。

## 8. 置信度 `confidence`

置信度不是固定值，而是根据以下因素动态计算：

- 有效样本量
- 历史覆盖时间跨度
- 当前 IP 的历史样本数
- 历史一致性

除此之外，PoW 还会进一步修正置信度。

### 8.1 一致时提高置信度

如果：

- IP 风险看起来高，PoW 也高风险
- 或 IP 风险看起来低，PoW 也偏正向

那么说明两条证据链一致，会提高置信度。

### 8.2 冲突时降低置信度

如果：

- IP 风险判断偏好，但 PoW 很差
- 或 IP 风险判断很差，但 PoW 偏强

那么说明信号冲突，会降低置信度。

这比“强行给出确定结论”更合理。

## 9. 最终判定逻辑

最终判定不是只看 `compositeRisk`，而是：

1. 先看当前 `IPPure` 实时分所在区间
2. 再看综合风险
3. 再看稳定性、趋势、异常模式
4. 最后让 PoW 信号做安全门或风险门

### 9.1 区间 `0-15`

只有在下面这些条件同时满足时，才会给出非常安全的结论：

- 综合风险足够低
- 稳定性高
- 趋势不恶化
- 最近没有高风险持续
- PoW 没有明显负面信号

### 9.2 区间 `16-25`

通常属于较优区间，但如果出现：

- 锯齿波动
- 高风险占比偏高
- PoW 持续偏低

则会从“安全”降到“观察”。

### 9.3 区间 `26-40`

这是算法最重点研判的灰区。

在这个区间内，算法会重点看：

- 当前 IP 历史是否稳定
- 最近是否恶化
- 是否存在锯齿 / 阶梯模式
- 是否出现连续高风险
- PoW 是否在持续偏低

也就是说：

> 同样是 `35` 分，有的节点可以继续用，有的节点已经危险了。

### 9.4 区间 `41-50`

进入明显风险区。  
如果再叠加：

- 异常模式
- 持续高风险
- PoW 持续低难度

通常会直接进入危险判断。

### 9.5 区间 `50+`

通常直接视为强风险区，建议更换节点。

### 9.6 PoW 的最终作用

PoW 在最终判定里主要做三件事：

1. 阻止“本该谨慎的节点”被过早判为安全
2. 在信号一致时增强当前结论
3. 在信号冲突时降低置信度

## 11. 如何正确理解结果

### 情况 A：当前分低，历史稳定，PoW 也高

这通常是比较理想的情况。  
说明：

- 当前 IP 风险低
- 历史走势稳定
- ChatGPT 入口质量也不错

结论通常会偏向安全。

### 情况 B：当前分一般，但历史越来越差，PoW 也低

这是最应该警惕的情况之一。  
说明：

- 当前还没彻底崩
- 但趋势已经恶化
- 且 PoW 也在给负面信号

这类节点常常正处于“正在变差”的阶段。

### 情况 C：当前分一般，历史不差，但 PoW 明显偏低

这类情况通常不会直接判死刑，但算法会：

- 提高综合风险
- 降低置信度
- 给出更保守的结论

因为这说明：

- IP 维度和 PoW 维度出现了冲突
- 需要更多样本来确认

## 12. 相比旧版的增强点

当前版本相对旧版，主要有这些增强：

1. 从“单点 IP 分数判断”升级为“时序风险分析”
2. 从“只看均值”升级为“均值 + 趋势 + 稳健统计 + 持续性风险”
3. 会区分“当前节点差”与“最近频繁换节点造成的数据混杂”
4. 引入 `PoW difficulty` 作为第二条证据链
5. 当 IP 和 PoW 一致时提高置信度，冲突时降低置信度
6. 更不容易被一次刷新或一次偶然低分带偏

---

## 13. 局限性与注意事项

这套算法已经比“单次看分”可靠得多，但仍然不是绝对真相。

需要注意：

### 13.1 PoW 不是唯一标准

PoW 会受页面状态、请求上下文、具体问题复杂度影响。  
因此它只能做辅助信号，不能单独作为“百分百降智”的依据。

### 13.2 样本量很重要

如果历史样本太少：

- 置信度会偏低
- 判断会偏保守

### 13.3 IPPure 可能存在回退情况

如果 `IPPure` 无法按真实出口 IP 查询，脚本会回退到直接查询。  
这时当前显示的 IP 分数可能和真实出口存在偏差。

### 13.4 脚本依赖网页接口

如果 ChatGPT 的前端结构、接口路径或字段发生明显变化，PoW 采集逻辑可能需要更新。

---

## 14. 一句话总结

这套算法的本质，不是在回答“你这次分数高不高”，而是在回答：

> 你当前这个 ChatGPT 会话入口，结合 IP 干净度、最近一段时间的历史行为，以及 PoW 难度信号，看起来像不像一个正在变差、容易降智的节点。

所以它更像一个“持续型风险判断模型”，而不是一个“单次读分器”。
